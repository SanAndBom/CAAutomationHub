방향성 요약

이 프로젝트는 UI prototype에서 출발해 Runtime canonical state, polling publish path, XGT reuse boundary, Pilot business flow, FLOW.JSON 전략으로 이동 중이다.

1. Runtime이 중심

Runtime은 canonical state의 중심이다. `RuntimeSnapshot`은 임시 DTO가 아니라 WPF와 운영 흐름이 공유하는 상태 계약이다.

2. UI는 표현 계층

WPF Dashboard는 `DashboardSnapshot`을 표시한다. polling, reconnect, driver detail은 UI 책임이 아니다.

3. Driver는 primitive

`XgtDriverCore`는 protocol, transport, recovery primitive에 집중한다. business flow를 직접 소유하지 않는다.

4. FLOW.JSON은 업무 정의

`FLOW.JSON`은 XGT command list가 아니라 PLC별 Business Flow Definition이다. 초기 구조는 `flow / bindings / metadata`다.

Boundary Map

문서들이 반복해서 지키는 핵심 경계다. 어느 작업을 하든 이 선을 넘지 않는 것이 ACCEPT의 조건이다.

01

WPF Dashboard

사용자에게 상태를 보여준다. Runtime 내부 polling과 driver detail을 직접 수행하지 않는다.

02

Adapter / Mapper

`RuntimeSnapshot`을 `DashboardSnapshot`으로 변환한다. UI 친화 표현 계층이다.

03

Runtime Core

canonical state, channel registry, snapshot publish path를 관리한다. vendor-neutral을 유지한다.

04

Adapter / Handler

XGT read/write, DB query, payload build, ACK/error write 실행 책임을 맡는다.

05

Driver / FakePlc

Driver는 protocol/transport primitive, FakePlc는 통신/장애 시나리오 검증 하네스다.

# AGENTS.md

이 문서는 AutomationHub Rebuild 프로젝트에서 Codex가 작업 시작 전 확인해야 하는 최상위 실행 계약이다.

목적은 단순 코드 생성이 아니라, 장기 프로젝트의 맥락, 검증 흐름, 하네스 원칙, Boundary 계약을 보존하는 것이다.

---

## 1. Historical Record Rule

CAAutomationHub Runtime 작업의 primary historical record는 `docs/harness/AH-*.md` Closeout 문서다.

ContextPublisher 자동 publish는 현재 사용하지 않는다.
다시 필요해지면 사용자 별도 요청 후 재도입한다.

다음 상황에서는 작업 결과를 Closeout 문서 또는 최종 보고에 남긴다.

* 코드 구조 변경 (refactor)
* 기능 추가 또는 수정
* 로직 흐름 변경
* 예외 처리 변경
* 테스트 보강 또는 시나리오 추가
* Runtime / Driver / WPF / Harness 경계에 영향을 주는 변경

## 2. Required Behavior

Codex는 코드 변경 작업을 완료한 후 변경 내용, 검증 결과, Harness / Boundary / Validation 영향을 보고해야 한다.

Closeout이 필요한 작업에서는 `docs/harness/AH-*.md` 문서에 historical record를 남긴다.

ContextPublisher 자동 publish, Implementation Event 자동 publish, Verification Event 자동 publish는 현재 작업 절차에 포함하지 않는다.

## 3. Relation to Verification

관련 테스트가 있다면 실행한다.

작업 완료 전에는 빌드 또는 테스트 필요성, 실행 명령과 결과, 변경 파일 목록, Harness / Boundary / Validation 영향, ACCEPT 판정 가능 여부를 확인해야 한다.

## 4. Cognitive Contract Reference

AutomationHub는 단순 기능 모음이 아니라 기존 PLC 중계 구조를 대체하는 장기 Runtime 플랫폼이다.

Long-term cognitive contract 문서:

* docs/context/META_IPRO_CODEX_COGNITIVE_INTERFACE.md
* docs/context/COGNITIVE_SYNC_CHECK.md

ContextPublisher automatic publish는 현재 disabled / not used 상태다.
Runtime 작업의 primary historical record는 `docs/harness/AH-*.md` Closeout 문서다.

## 5. Core Invariants

* FlowTest는 독립 실행자가 아니다.
* FlowTest / Runtime / Simulation은 가능한 한 동일한 실행 경로를 공유해야 한다.
* Runtime shared execution path를 깨뜨리면 안 된다.
* Runtime은 canonical state의 중심이다.
* UI는 Runtime 내부 polling, reconnect, driver 세부 책임을 침범하지 않는다.
* Driver는 protocol / transport / recovery primitive 책임에 집중한다.
* Supervisor는 재연결, 회복, 균형, 운영 안정성의 상위 조정자다.
* FakePlc는 단순 mock이 아니라 통신/장애 시나리오 검증 하네스다.
* Harness 없는 완료는 완료가 아니다.
* Verification evidence 없는 완료는 완료가 아니다.
* ACCEPT는 단순 기능 성공이 아니라 의미, 경계, 검증이 유지되었음을 뜻한다.

## 6. Work Instruction Shape

Codex 작업 지시문은 가능하면 Goal, Context, Constraints, Done when 구조를 따른다.

AutomationHub 작업에서는 Harness, Boundary, Validation, ACCEPT 흐름도 함께 보존한다.

## 7. Boundary Awareness

Runtime, WPF Dashboard, FlowTest, FakePlc, XgtDriverCore, Supervisor 관련 작업 전에는 boundary 영향을 점검한다.

불확실하면 구현하지 말고 Boundary Review로 전환한다.

## 8. Cognitive Sync Awareness

새 세션은 `docs/context/COGNITIVE_SYNC_CHECK.md`의 Cognitive Checksum을 통해 맥락 복원 여부를 검증한다.

## 9. Memory Hygiene Rules

확인된 계약만 Core Invariant에 기록한다.
불확실한 내용은 확인 필요로 표시한다.
임시 아이디어와 확정 계약을 분리한다.
오래된 정보가 현재 작업을 오염시키지 않도록 주의한다.
